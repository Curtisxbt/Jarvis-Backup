# 2026-03-15

## Decisions
- Denis a validé une refonte complète du système mémoire Jarvis et a demandé une implémentation globale immédiate, sans fragmentation en multiples itérations.
- Décision finale prise avec Denis : après l’implémentation du nouveau pipeline mémoire, **ne plus toucher à l’architecture pour l’instant** ; laisser vivre le système tel quel puis observer les résultats réels avant toute nouvelle modification.

## Facts learned
- Le vrai problème mémoire identifié n’était pas l’absence de structure, mais la **faible qualité du pipeline d’écriture**, surtout les fichiers journaliers trop vagues et trop pauvres.
- Denis valide le principe suivant : le **daily file n’est pas la mémoire finale**, mais une capture journalière exploitable servant de matière première à la consolidation.
- Denis a explicitement refusé l’ancienne “étape 3” type score de fiabilité ; la trajectoire retenue est : structure minimale, prérequis/modes d’échec, post-mortem, amendements contrôlés, puis pipeline mémoire strict.

## User preferences confirmed
- Denis veut une mémoire long terme **très stricte** pour éviter qu’elle soit polluée au bout de dizaines de semaines.
- Denis préfère une logique de **promotion minimale**, anti-bruit, anti-duplication, avec refus des formulations vagues.
- Denis veut une **consolidation hebdomadaire** le lundi à 02:00 Europe/Paris et un **nettoyage mensuel** le premier lundi du mois à 02:30 Europe/Paris.
- Denis a validé qu’après cette implémentation, on **fige** le système et on n’ajoute rien de plus tant qu’on n’a pas observé plusieurs cycles réels.

## Errors / frictions
- Les anciens cron du soir étaient structurellement trop faibles : prompts de “passe courte / signaux évidents”, ce qui poussait mécaniquement à écrire des clôtures pauvres et peu exploitables.
- Le risque principal identifié sur la mémoire était la **pollution progressive** du long terme en l’absence de filtre temporel, taxonomie de promotion et revue mensuelle conservatrice.

## Lessons learned
- Un bon système mémoire ne se joue pas seulement sur les fichiers, mais sur la chaîne complète : **capture journalière utile → coordination partagée → promotion durable stricte → consolidation hebdo → review mensuelle conservatrice**.
- Pour éviter d’empirer le système, il faut s’arrêter après une bonne architecture et **observer le comportement réel** avant d’ajouter d’autres couches.

## Promotion candidates
- Protocol : le pipeline mémoire Jarvis repose désormais sur 4 couches explicites : daily privé, daily shared, memory long terme privé, memory long terme shared.
- Protocol : chaque item journalier doit satisfaire une vraie valeur minimale (utile à un futur run, décision, préférence/contrainte, friction/leçon, open loop réel, utilité cross-agent, ou candidat crédible à promotion).
- Protocol : toute promotion long terme doit passer un filtre 1 mois / 3 mois et entrer dans une taxonomie explicite (`Preference`, `Constraint`, `Decision`, `Protocol`, `System lesson`, `Long-term project fact`, `Cross-agent shared fact`).
- Protocol : un cron hebdomadaire `weekly-memory-consolidation` a été ajouté pour consolider les 7 derniers fichiers journaliers disponibles.
- Protocol : un cron mensuel `monthly-memory-review` a été ajouté pour nettoyer de façon conservatrice les mémoires long terme le premier lundi du mois à 02:30 Europe/Paris.
